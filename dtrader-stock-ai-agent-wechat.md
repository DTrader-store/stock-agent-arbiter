# 用 DTrader 写一个股票 AI Agent：让大模型先学会“看证据”

如果要做一个股票 AI Agent，很多人的第一反应是：

> 输入一只股票，让 AI 告诉我能不能买。

这个方向听起来直接，但其实很危险。

股票分析最怕的不是模型“不够聪明”，而是模型在没有数据边界的情况下侃侃而谈。它可能把市场传闻、历史印象、模糊判断混在一起，最后生成一段看似专业、却很难复查的结论。

所以我更倾向于从另一个问题开始：

> 能不能让 AI 只基于真实行情数据，先把一只股票的正反证据讲清楚？

这就是 `stock-agent-arbiter` 这个项目的出发点。

它不是一个自动交易系统，也不会替你下单，更不会输出“买入”“卖出”这样的交易指令。它更像一个股票研究助手：用 DTrader V3 获取行情数据，让 AI 扮演买方、卖方和仲裁者，围绕同一份数据完成一场结构化辩论，最后生成一份可复查的 Markdown 报告。

## 为什么要用 DTrader 做数据底座

股票 AI Agent 的第一层能力，不应该是“会说”，而应该是“有据可查”。

DTrader V3 提供的是行情和市场数据接口，这非常适合做 Agent 的底层数据源。因为它能让模型面对的不是一个开放问题，而是一份明确的数据快照。

在这个项目里，Agent 会读取这些只读数据：

- `quote`：报价数据
- `kline`：日 K 数据
- `details`：成交明细
- `trends`：分时趋势
- `finance`：财务相关数据
- `ranking_stocks`：股票榜单
- `ranking_boards`：板块榜单

这些数据会被整理成一份统一的 `EvidenceSnapshot`。后面的所有 AI 分析，都只能围绕这份快照展开。

这一步看似普通，但它决定了整个 Agent 的质量：模型不是凭感觉分析股票，而是在一个受控的数据环境里做证据整理。

## 这个 Agent 的核心玩法：买方、卖方、仲裁者

项目没有让大模型一次性输出最终结论，而是把分析过程拆成三个角色。

第一个角色是买方。

买方只负责从同一份 DTrader 数据里提取偏积极的证据。比如价格变化、成交情况、财务指标、榜单位置等，只要数据中能支持积极观点，就把它整理出来。

第二个角色是卖方。

卖方站在谨慎视角，同样只能引用这份数据，寻找风险点、弱信号或证据不足的地方。

第三个角色是仲裁者。

仲裁者不会说“应该买”或“应该卖”，它只做一件事：总结双方证据强弱、列出争议点、指出证据缺口，并给出一个保守摘要。

这样设计的好处是，AI 不再是一个黑盒判断器，而是被放进了一个可观察、可追踪的研究流程里。

## 工作流是怎么跑起来的

项目用 LangGraph 编排整个流程。

主流程大致是这样：

```text
collect_data
  -> validate_data
  -> buyer_evidence
  -> seller_evidence
  -> debate
  -> arbiter
```

每个节点都只做一件事。

`collect_data` 负责从 DTrader V3 获取市场数据。

`validate_data` 负责检查数据是否可用。

`buyer_evidence` 和 `seller_evidence` 分别生成买方和卖方证据。

`debate` 会让双方进行三轮辩论。

`arbiter` 最后输出结构化仲裁结果。

这里有一个很实用的细节：如果部分数据源不可用，流程不会直接中止。

比如行情服务里 `ranking_stocks` 或 `ranking_boards` 暂时不可用，但 `quote`、`kline`、`finance` 仍然能正常返回，那 Agent 会继续使用已有数据完成分析，同时把失败来源记录到“证据缺口”里。

只有当所有数据源都不可用时，流程才会中止，避免模型在没有事实基础时硬编结论。

这也是做金融类 AI Agent 时非常重要的一条原则：

> 可以接受数据不完整，但不能让模型假装数据完整。

## Prompt 的关键约束

这个项目里的 prompt 不是为了让模型“更会说服人”，而是为了限制模型不要越界。

买方 prompt 明确要求：

- 只能基于用户提供的 V3 数据快照
- 不得编造外部事实
- 不给交易建议
- 输出 3-5 条证据，并标注数据来源

卖方 prompt 也是同样逻辑，只是立场换成谨慎视角。

辩论 prompt 要求双方只能引用共享数据和已有证据，不能引入外部事实，也不能输出交易指令。

仲裁 prompt 更进一步：要求模型严格输出 JSON，并包含固定字段：

```json
{
  "buyer_strongest_evidence": ["..."],
  "seller_strongest_evidence": ["..."],
  "disputed_points": ["..."],
  "evidence_gaps": ["..."],
  "conservative_summary": "...",
  "disclaimer": "本报告不构成交易建议。"
}
```

这套约束让结果更容易被程序解析，也更容易被人复查。

## 终端展示和报告生成

运行命令很简单：

```bash
uv run stock-agent-arbiter 600519
```

运行后，终端会用 Rich 展示完整过程：

- V3 数据快照状态
- 哪些来源可用
- 哪些来源缺失
- 买方证据
- 卖方证据
- 三轮辩论
- 仲裁汇总
- Markdown 报告路径

报告会保存到：

```text
reports/{code}-{timestamp}.md
```

这份报告不是一句简单的“看多”或“看空”，而是完整记录了数据状态、证据提取、双方争议和最终摘要。

对研究流程来说，这比一个无法追溯的结论更有价值。

## 项目结构并不复杂

整个项目保持得很轻量。

核心模块包括：

- `cli.py`：命令行入口，负责读取配置、启动工作流、写报告
- `config.py`：读取 DTrader 和 OpenAI 相关环境变量
- `v3_adapter.py`：封装 DTrader V3 只读数据访问
- `workflow.py`：LangGraph 工作流核心
- `prompts.py`：买方、卖方、辩论、仲裁 prompt
- `models.py`：Pydantic 数据模型
- `renderer.py`：Rich 终端渲染
- `report.py`：Markdown 报告生成

依赖也很直接：

- `dtrader-v3-sdk`
- `langgraph`
- `langchain-openai`
- `pydantic`
- `typer`
- `rich`
- `python-dotenv`

这意味着它不是一个庞大的平台工程，而是一个可以快速理解、快速改造的 Agent 原型。

## 这个项目真正有价值的地方

我认为它最有价值的地方，不在于“AI 能不能预测股价”。

真正有价值的是，它展示了一种更可靠的股票 AI Agent 设计方式：

1. 数据先行，而不是 prompt 先行。
2. 只读分析，而不是直接交易。
3. 多角色对抗，而不是单次生成结论。
4. 显式记录证据缺口，而不是掩盖数据不足。
5. 输出结构化结果，而不是只生成一段漂亮文字。
6. 最终报告可复查，而不是一次性聊天记录。

这类 Agent 的定位不应该是“替你做决定”，而应该是“帮你把证据摆清楚”。

在金融场景里，这个边界非常重要。

## 后续可以怎么扩展

这个项目现在还是 CLI 版本，后续可以继续增强：

- 增加更严格的股票代码规则
- 支持更多 DTrader V3 数据源
- 支持批量股票分析
- 增加 JSON 导出
- 增加 Web UI
- 增加报告对比功能
- 增加不同研究模板
- 加入模型输出质量评估
- 加入真实接口集成测试

但不管怎么扩展，我认为底层原则都应该保持不变：

> 让 AI 基于数据说话，让过程可以复查，让结论保持克制。

## 结语

用 DTrader 写股票 AI Agent，不一定要一开始就奔着自动交易去。

更实际、也更稳妥的起点，是先做一个只读的研究 Agent：它读取真实行情数据，组织多角色分析，暴露证据缺口，最后生成一份可复查的报告。

这也是大模型在金融研究场景里更适合的位置。

它不是替代判断的人，而是一个更快整理证据、更稳定执行流程、更容易留下记录的研究助手。

当 AI 不再急着给答案，而是先把证据讲清楚，它才真正开始变得有用。
